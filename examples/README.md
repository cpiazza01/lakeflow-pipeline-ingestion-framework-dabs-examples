# Example: Incremental Ingestion from a Databricks Volume

This example demonstrates two complementary incremental ingestion patterns using [Synthea](https://github.com/synthetichealth/synthea) synthetic patient CSV files that land in a **Unity Catalog Volume**. The `raw_files` Volume is provisioned by the bundle — no manual infrastructure setup required beyond the catalog and `staging` schema already existing.

Sample data for three simulated daily loads is included in [sample-data/](./sample-data/).

## What This Example Shows

### Pattern 1 — SCD Type 1 (Incremental Upsert)
**Used for:** Dimension-style data where records change over time and you only need the current state.

Source: Synthea `patients.csv`. Each daily file is a full patient snapshot. Records are upserted into `silver.patients` using `Id` as the CDC key. `_metadata.file_modification_time` is used as the sequence column — it reflects when the file was written to the Volume, so it remains correct even across full pipeline reloads. SSN and address are masked via Unity Catalog masking functions.

### Pattern 2 — Streaming (Append-Only)
**Used for:** Fact/event data that is naturally immutable — each file adds new records and nothing is ever modified.

Source: Synthea `encounters.csv`. Each daily file contains a new batch of encounter records appended directly to `silver.encounters`. This table joins to `silver.patients` on `member_id` and serves as the primary fact table for downstream DBT gold layer models.

## Volume Structure

The `raw_files` Volume is created by the bundle inside the pre-existing `staging` schema. Drop each dated sample-data subfolder into the corresponding Volume path to simulate daily arrivals:

```
/Volumes/<catalog>/staging/raw_files/
├── patients/                  <- SCD1 source; daily Synthea patient snapshot CSVs
│   ├── 2024-01-01/
│   │   └── patients.csv       (70 patients — initial load)
│   ├── 2024-02-01/
│   │   └── patients.csv       (75 patients — 8 updated + 5 new)
│   └── 2024-03-01/
│       └── patients.csv       (106 patients — 6 more updated + 31 new)
└── encounters/                <- Streaming source; daily Synthea encounter CSVs
    ├── 2024-01-01/
    │   └── encounters.csv     (~1,643 records)
    ├── 2024-02-01/
    │   └── encounters.csv     (~1,643 records)
    └── 2024-03-01/
        └── encounters.csv     (~1,644 records)
```

> The `staging` schema and catalog must already exist. Only the `raw_files` volume is provisioned by the bundle.

## How to Deploy

```bash
# Install the framework (first time only)
pip install git+https://github.com/cpiazza01/lakeflow-pipeline-ingestion-framework-dabs.git

# Generate the Lakeflow Declarative Pipelines SQL and DABs resource files
lakeflow-generate --config pipeline_config.yaml --env dev

# Deploy everything — including the Volume — in one command
databricks bundle deploy --target dev
```

## Key Configuration Decisions

| Setting | Value | Why |
|---|---|---|
| `source_path` | `/Volumes/.../raw_files/patients` | Volumes are the recommended way to land files for Lakeflow Declarative Pipelines in Unity Catalog workspaces |
| `source_file_type` | `csv` | Synthea outputs CSV natively — no conversion step needed |
| `table_type: scd1` | Patients | Records change over time; only current state needed |
| `table_type: streaming` | Encounters | Encounters are immutable once recorded; append-only |
| `sequence_by` | `_metadata.file_modification_time` | Set at file write time — stays correct on full pipeline reloads |
| `schedule` | Daily cron | Volumes do not support file-arrival triggers; cron aligns with daily file delivery |
| `column_masks` | SSN, ADDRESS | PII masked at the silver layer via Unity Catalog masking functions |
| `expectations` | On key columns | Fail early on missing keys rather than silently loading nulls |
