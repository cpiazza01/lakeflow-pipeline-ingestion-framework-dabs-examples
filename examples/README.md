# Example: Incremental Ingestion from a Databricks Volume

This example demonstrates two complementary incremental ingestion patterns using [Synthea](https://github.com/synthetichealth/synthea) synthetic patient CSV files that land in a **Unity Catalog Volume**. The `raw_files` Volume is provisioned by the bundle вЂ” no manual infrastructure setup required beyond the catalog and `staging` schema already existing.

Sample data for three simulated daily loads is included in [sample-data/](./sample-data/).

## What This Example Shows

### Pattern 1 вЂ” SCD Type 1 (Incremental Upsert)
**Used for:** Dimension-style data where records change over time and you only need the current state.

Source: Synthea `patients.csv`. Each daily file is a full patient snapshot. Records are upserted into `silver.patients` using `Id` as the CDC key. `_metadata.file_modification_time` is used as the sequence column вЂ” it reflects when the file was written to the Volume, so it remains correct even across full pipeline reloads. SSN and address are masked via Unity Catalog masking functions.

### Pattern 2 вЂ” Streaming (Append-Only)
**Used for:** Fact/event data that is naturally immutable вЂ” each file adds new records and nothing is ever modified.

Source: Synthea `encounters.csv`. Each daily file contains a new batch of encounter records appended directly to `silver.encounters`. This table joins to `silver.patients` on `member_id` and serves as the primary fact table for downstream DBT gold layer models.

## How to Deploy

```bash
# Install the framework (first time only)
pip install git+https://github.com/cpiazza01/delta-live-tables-ingestion-framework-dabs.git

# Generate the DLT SQL and DABs resource files
dlt-generate --config pipeline_config.yaml --env dev

# Deploy everything вЂ” including the Volume вЂ” in one command
databricks bundle deploy --target dev
```

## Key Configuration Decisions

| Setting | Value | Why |
|---|---|---|
| `source_path` | `/Volumes/.../raw_files/patients` | Volumes are the recommended way to land files for DLT in Unity Catalog workspaces |
| `source_file_type` | `csv` | Synthea outputs CSV natively вЂ” no conversion step needed |
| `table_type: scd1` | Patients | Records change over time; only current state needed |
| `table_type: streaming` | Encounters | Encounters are immutable once recorded; append-only |
| `sequence_by` | `_metadata.file_modification_time` | Set at file write time вЂ” stays correct on full pipeline reloads |
| `schedule` | Daily cron | Volumes do not support file-arrival triggers; cron aligns with daily file delivery |
| `column_masks` | SSN, ADDRESS | PII masked at the silver layer via Unity Catalog masking functions |
| `expectations` | On key columns | Fail early on missing keys rather than silently loading nulls |
